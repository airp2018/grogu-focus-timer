# Grogu 电子宠物扩展 — 设计需求 v2

> 基于 v1 评审反馈整理。产品定位：**电子宠物伴随，不是时间管理工具**。
> 古古是中场休息的调节，不是主角；随机出现带来愉悦，提醒专注是顺带的。

---

## 一、产品定位（锁定）

1. **不做什么**：不做成长系统、不做等级、不做连续天数、不做打卡。
   理由：成长意味着责任，会从"愉悦感"变成"负担感"，违背产品初衷。
2. **做什么**：随机出现的小东西，瞥一眼会笑，调节一下，回去干活。
3. **主角是用户**：古古是配角。它分散注意力是 bug，不是 feature。
   "中场休息"四个字就是它存在的全部理由。

---

## 二、八条迭代方向（按用户标注的优先级）

### 1. idle 常驻态 — 不做

**用户决定**：不要 idle 常驻。古古不主动趴在屏幕角。
- 理由：它会分散注意力，反客为主。
- 现有方案已对：事件触发才出现，结束就消失。
- 真正要补的是"随机出现的频次和时机"，不是"一直在那"。
- 提醒专注是顺带的，不是古古的主要职能。

### 2. 行为谱扩展 — 进行中

**用户决定**：认同当前行为谱过窄（全是"吃"），继续找表情包补充。
- 待办：补充非"吃"类素材，例如：
  - 翻身 / 打哈欠
  - 看你一眼又睡回去
  - 伸懒腰
  - 玩原力球
  - 深夜打盹 zzz
- 素材来源：剧集截图、表情包采集。
- 每个新素材需要 .webp + .m4a 一对（沿用现有 assets/ 结构）。

### 3. 鼠标互动 — 优先考虑 ⭐

**用户决定**：认同点击反应单一，先做这一项。
- 当前：`content.js` 第 120-122 行 `createForceParticles` 永远喷 15 个粒子。
- 要做：让鼠标交互有"手感"。具体方向待定，候选：
  - **悬停**：鼠标靠近时古古微微反应（抬头 / 转头看鼠标方向）
  - **点击位置**：点头部 = 摸头（眯眼享受）；点身体 = 戳（嫌弃躲开）；连续点击 = 烦
  - **拖拽**：可拖动到屏幕任意位置
  - **长按**：被抱起，挣扎或安静
- 设计约束：互动后必须能自动回到原位 / 自动消失，不能让用户分心去"照料"它。
- 这是下一轮要细化的重点。

### 4. NO blocker 文案 — 不改

**用户决定**：维持现状。
- "Mando, 快回去工作！这不是正道！" 是剧集梗的幽默管制，不是真管教。
- 大部分用户不会觉得被冒犯，反而觉得好笑。
- 保持原汁原味的曼达洛人设定。

### 5. 场景化随机事件 — 细化中 ⭐

**用户决定**：把 `task` 字段从"死配置"改成"场景触发器"。
- 当前：`popup.html` 第 49 行让用户填"我当前要做的任务"，但加载后从不用于互动。
- 改造方向：
  - 用户在配置里选/填当前场景，例如："追剧"、"写代码"、"看论文"、"摸鱼"。
  - 不同场景触发不同的随机事件子集。
  - 举例：设了"追剧"，古古会在追剧过程中冒出来吃爆米花、打瞌睡、跟着看屏幕。
  - 举例：设了"写代码"，古古会在你卡住时歪头看屏幕、打哈欠暗示该休息。
- 这把"随机出现"从纯随机变成"和当前场景绑定的惊喜"，愉悦感密度更高。
- 待办：定义场景列表 + 每个场景对应哪些素材。

### 6. 劝阻措辞 — 不改

**用户决定**：维持"劝阻"概念。
- 理由：这是"小孩管大人"的反差萌，不是真训斥，没人会反感。
- 反感可以关闭声音（已有 `sound-toggle`），这是最低干扰的兜底。
- 不需要把"劝阻"重命名成"陪伴反应"之类——那反而丢掉了反差幽默。

### 7. auto-dismiss 时长 — 维持，加限额 ⭐

**用户决定**：维持自动消失机制。
- 理由：主人要干活，古古不能赖着不走。
- 看不够可以主动召唤（快捷键 / popup 按钮），每次召唤换一个表情。
- **新增**：每天主动召唤设限额（例如 N 次/天）。
  - 理由：避免用户沉迷逗宠物，本末倒置。
  - 限额到顶后，召唤按钮置灰 + 提示"古古今天累啦，明天再来"。
  - 随机事件不受限额影响（那是被动的，不算用户"主动消耗"）。
- 待办：决定每日限额数值、限额重置时机（午夜 / 安装周年）。

### 8. 状态叠加 — 合并到 #3

**用户决定**：本质还是鼠标互动问题。
- 当前 `replaceChildren` 互斥是合理的（避免视觉混乱）。
- 真正想要的是"古古在吃马卡龙时你摸它，它会边吃边看你"——这需要的是**对当前正在播放的状态做交互响应**，不是同时显示两个独立状态。
- 合并到 #3 鼠标互动方案里一起设计。

## 四、约束（所有迭代必须遵守）

1. 古古不主动分散注意力。任何"持续显示"都要可关闭、可降级。
2. 默认最低干扰：声音可关、视觉可关。
3. 不引入任何"养成 / 成长 / 打卡"机制。
4. 任何新增素材必须沿用 `assets/{name}.webp + {name}.m4a` 结构。
5. 状态机改动不要破坏现有快捷键召唤 / 随机事件 / blocker 三条链路。

## 五、v2.1 产品收敛：从"时间管理"改成"分心守门员 + 古古陪伴"

本轮讨论后，产品边界进一步收窄：

1. **不做工作判官**
   - 插件不判断 GitHub、Claude、localhost、搜索引擎是不是"正在工作"。
   - 非黑名单页面一律视为中性，不评价、不夸奖、不打扰。
   - `YES` 不代表"你在工作"，只代表"你离开了明确分心页面"。

2. **只明确识别 NO**
   - 产品目标是识别明确分心，而不是猜测正确行为。
   - 命中分心黑名单才触发 `NO`。
   - 从黑名单切换到非黑名单，触发一次轻量 `YES`。
   - 黑名单之外保持安静。

3. **离开浏览器默认静默**
   - 切到 IDE、Word、终端、设计软件都可能是正事。
   - 不再因为离开浏览器触发 `NO`。
   - 如果未来需要"离开浏览器提醒"，必须是独立开关，默认关闭。

4. **去掉休息模式**
   - "休息时长"不是核心价值。
   - 专注结束后提示一次，然后回到下一轮专注待开始。
   - 不自动进入 break 倒计时。
   - 古古是中场调节，不是番茄钟教练。

5. **暂停不是核心概念**
   - 产品上不强调"暂停"，因为它不是严格计时工具。
   - 如果保留暂停按钮，只作为技术兜底，而不是主流程。
   - 主流程应该是：开始一段专注陪伴 -> 期间守住黑名单 -> 结束后停下。

## 六、分心黑名单模型（替代原工作关键词白名单）

### 1. 判断规则

浏览器内页面只分两类：

| 类型 | 条件 | 行为 |
|------|------|------|
| 分心页面 | URL 或标题命中黑名单 / 正则 | 触发 `NO` blocker |
| 中性页面 | 不命中黑名单，或浏览器系统页 | 不判断，不打扰 |

状态转移：

| 上一状态 | 新页面 | 行为 |
|---------|--------|------|
| 非分心 | 分心 | `NO` |
| 分心 | 分心 | 保持 blocker，不重复轰炸 |
| 分心 | 非分心 | `YES` 一次 |
| 非分心 | 非分心 | 静默 |

### 2. 默认黑名单分组

默认黑名单不是全部强制开启，而是给用户可勾选的分组。

| 分组 | 默认 | 示例 |
|------|------|------|
| 视频/短剧 | 开 | YouTube、Bilibili、抖音、快手、爱奇艺、优酷、腾讯视频、芒果 TV、西瓜视频、红果短剧、短剧站 |
| 社交媒体 | 开 | 微博、X/Twitter、Instagram、Facebook、Threads |
| 短视频关键词 | 开 | shorts、reels |
| 游戏/直播 | 关 | Steam、Epic、Twitch、斗鱼、虎牙、TapTap、4399 |
| 论坛热点 | 关 | 贴吧、知乎热榜、Reddit |
| 购物网站 | 关 | 淘宝、京东、拼多多、Amazon |

设计理由：

- 视频 / 短剧 / 社交 / 短视频是最稳定的高分心类别，默认开启。
- 游戏/直播、论坛和购物因人而异，默认不勾选，避免误杀。
- 用户可以自行增删正则，产品不替用户定义人生。

### 3. 自定义规则

设置里保留一个文本框：

```text
youtube.com
bilibili.com
.*shorts.*
weibo.com
```

规则：

- 一行一个，也允许逗号分隔。
- 作为正则匹配 URL 和标题。
- 正则写错时忽略该行，不让插件崩。
- 自定义规则与默认分组取并集。

## 七、伴随式随机事件：有节奏，不是真随机

随机事件的目标不是"惊吓"，而是"伴随"。

### 1. 分布原则

不再使用固定 `5-10 分钟随机一次`。

改成按专注时长生成出现点：

| 专注时长 | 建议出现次数 | 建议分布 |
|----------|--------------|----------|
| 5 分钟 | 1 次 | 约 60%-75% 处 |
| 10 分钟 | 2 次 | 约 35%、75% |
| 25 分钟 | 4 次 | 约 20%、45%、68%、88% |
| 45 分钟 | 5-6 次 | 均匀分布但带轻微浮动 |

### 2. 浮动规则

- 每个出现点增加少量随机浮动，例如前后 8%-12%。
- 开始后前 60 秒不出现。
- 结束前 30-60 秒不出现。
- 如果专注时间很短，至少保证一次出现。

### 3. 场景关联

`task` 字段不参与工作判断，只用于选择随机事件语气和素材池。

示例：

| 场景 | 事件倾向 |
|------|----------|
| 写代码 | 歪头看屏幕、打哈欠、原力球、小声提醒 |
| 追剧 | 吃爆米花、一起看屏幕、打瞌睡 |
| 看论文 | 眯眼、思考、趴下、困了 |
| 摸鱼 | 更容易出现 NO 风格的小管教 |

如果没有足够素材，先退回通用素材池。

## 八、侧边栏 UI 信息架构

侧边栏不是设置面板堆叠，而是"专注驾驶舱"。

### 1. 主界面

主界面只放高频操作：

1. 顶部：Grogu Focus + 设置齿轮。
2. 中间：当前专注倒计时。
3. 控制：开始 / 重置；暂停如果保留，应弱化。
4. 召唤古古：标题为"召唤古古"，说明为"如果想它，就记得召唤一下吧。"

### 2. 设置面板

设置应收纳低频项：

- 专注时长。
- 当前场景 / 任务。
- 默认分心黑名单勾选。
- 自定义分心关键词 / 正则。
- 声音提醒开关。
- 分心劝阻开关。

### 3. 状态显示

不要用"原力进行中"占右上角主视觉。

右上角改为设置齿轮；状态放在计时器附近，用轻量文字表达：

- 未开始
- 专注中
- 已完成

## 十、当前版本验收口径

一个版本是否符合 v2.1，按以下标准判断：

1. 打开 GitHub 不会自动 `YES`，除非它是从黑名单页面切回来的下一站。
2. 打开 YouTube / Bilibili 等默认黑名单会 `NO`。
3. 从 YouTube 切到任意非黑名单页面会 `YES` 一次。
4. 离开浏览器不会 `NO`。
5. 专注结束后不会自动进入休息倒计时。
6. 用户能勾选默认黑名单，也能写自定义正则。
7. 关闭声音后，表情包仍出现，但不播放音频。
